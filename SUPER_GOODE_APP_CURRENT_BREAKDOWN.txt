SUPER GOODE APP CURRENT BREAKDOWN
Updated: 2026-03-28

SCOPE
This breakdown is APP ONLY.
It covers the Super Goode mobile app built with Expo, React Native, and TypeScript.
The static Super Goode web map/site is excluded from this breakdown except where it matters as the app's canonical data source and remote feed relationship.

1. PROJECT OVERVIEW

Super Goode App is the native mobile companion to Super Goode Map.
It gives users a mobile-first way to browse the Super Goode restaurant list, view scores, open review videos, get directions, and save favorites.
The app is built around the shared Super Goode restaurant dataset and the current presentation is intentionally polished for iPhone and Android.

The app solves the mobile discovery problem for the brand:
- quick restaurant browsing on a phone
- a focused map/list experience
- direct access to reviews and directions
- local favorites that persist on the device

Relationship to the Super Goode brand and data:
- the web repo remains the canonical restaurant data source
- the app owns the mobile UI, navigation, filtering, review viewer, and status presentation
- the app does not own a separate permanent restaurant database
- the app can cache a validated remote snapshot on device, but that snapshot is runtime storage, not a new source of truth

2. CURRENT APP FUNCTIONALITY

Main tabs:
- Map
- Reviews
- Favorites
- Profile

Map tab:
- displays restaurant pins on a native map on iPhone and Android
- supports text search by name, city, or dish
- supports score filters
- requests foreground location permission and can center on the user's position
- shows a selected restaurant popup card when a pin is chosen
- provides actions for Review Video and Directions
- includes fit-to-restaurants and locate controls
- has a score-tier legend when nothing is selected

Reviews tab:
- shows a ranked feed of restaurant cards
- supports search and score filters
- includes a branded header and summary counters
- each card can open Directions or Watch Review
- query filtering is immediate while the displayed query value is deferred for smoother list behavior

Favorites tab:
- stores favorite restaurant IDs locally on the device
- restores favorites on launch
- filters saved items by search and score
- shows saved spot counts and current filtered counts
- uses the same restaurant cards as the reviews feed

Profile tab:
- shows current data-mode status
- shows whether the app is reading the local seed, live remote feed, cached remote snapshot, or bundled fallback
- shows coverage and local save counts
- keeps the app's settings runway clearly separate from current functionality

Review viewer:
- opens from restaurant cards, the Map popup, and the modal review route
- loads the selected review in an in-app WebView on native devices
- shows loading, missing review, blocked/failed, and external fallback states
- can open the review externally if the embedded view is not usable

Routes and deep links:
- the app has a modal review route at /review/[id]
- there is no live /restaurant/[id] or restaurant detail route
- legacy restaurant route handling is redirected away in the not-found screen for compatibility

Login and review submission limitations:
- there is no user login system in the app
- there is no native review submission flow
- the app opens existing review URLs instead of collecting user-authored reviews
- provider pages can still block embedded playback, which is why the fallback action exists

4. DATA FLOW / SOURCE OF TRUTH

Canonical source of truth:
- the web repo file ../Super Goode/data/locations.json remains the restaurant data source of truth
- the app never owns a separate permanent restaurant database

Local app seed:
- src/data/seed/locations.json is the app's current bundled parity snapshot
- the seed is meant to stay aligned with the source-of-truth dataset

Developer sync workflow:
- scripts/sync-locations-seed.mjs copies the web data into the app seed after confirming the source JSON parses
- this is a repo workflow, not the runtime cache

Runtime priority:
1. live remote feed
2. cached remote snapshot
3. bundled local seed fallback

Runtime cache details:
- src/services/restaurantRepository.ts loads the local seed by default when no remote feed is configured
- if EXPO_PUBLIC_LOCATIONS_FEED_URL is set, the app attempts to load a remote JSON feed first
- remote data is validated before acceptance
- accepted remote data is saved on device as a cached remote snapshot
- if the live feed fails or becomes invalid later, the app will try the cached remote snapshot next
- if neither live remote data nor a cached remote snapshot is available, the app falls back to the bundled local seed
- the cache lives in the app sandbox as super-goode-remote-snapshot.json and is not the same thing as the repo seed

Current feed modes and profile/status states:
- local-seed: no remote URL is configured, so the app is reading the bundled seed
- remote: the live feed loaded successfully and the app cached the accepted snapshot
- cached-remote: the live feed was unavailable or invalid, but a valid on-device snapshot exists
- local-fallback: the live feed and cached snapshot were unavailable, so the bundled seed is being used
- Profile can honestly report all of those states, including the cached snapshot timestamp when present

Current dataset shape used by the app:
- name
- score
- subtitle
- address
- city
- state
- lat
- lng
- directionsUrl
- reviewUrl
- sourceType
- confidence
- notes

Current verified data snapshot:
- 220 restaurants

- 1 low-confidence entry
- Daruma Restaurant is present in the current app seed
- the app seed reflects the latest verified shared dataset snapshot used by the app

Review URL handling:
- review URLs are normalized in-app before playback or fallback
- the stored seed mirrors the web source data, while runtime normalization keeps playback links consistent

9. KNOWN CONSTRAINTS AND CAVEATS

- No restaurant login or auth system exists yet
- No native review submission flow exists yet
- Favorites are local-only and will not sync across devices unless a future feature adds that
- Review provider pages can still block embedded playback or show login walls, so in-app playback is not guaranteed for every URL
- The app depends on the web repo for canonical restaurant data
- Any future source-of-truth edits that change restaurant names or addresses can affect derived restaurant IDs
- The app currently requests foreground location only; it does not do background location tracking
- Android release packaging is not fully finalized because app.json does not yet define an Android package name
- A launch with no EXPO_PUBLIC_LOCATIONS_FEED_URL configured will legitimately report Local seeded JSON active; that is not a cache failure
